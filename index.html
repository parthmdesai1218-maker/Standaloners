5 · Strategic Pillars
🏛️

The five pillars guiding every product decision

When the team debates what to build next, these pillars act as the tie-breaker — features that serve all five get priority. Features that conflict with even one get deprioritised.

Pillar 01
🧒 Child Safety First
Every feature is designed with an 8-year-old user in mind. No data collection from children, no direct-to-child advertising, parent gate for any adult action. Compliance with COPPA (US), GDPR-K (EU), and DPDP Act (India) is non-negotiable at every phase.
Pillar 02
📸 Content Quality Over Quantity
We will ship fewer species with outstanding photography and verified facts than flood the app with low-quality content. Every image is hand-curated. Every fact is sourced and attributed. This quality signal is what builds parent trust and teacher credibility.
Pillar 03
⚡ Instant Access, Zero Friction
The product must work the moment someone opens a link — no app install, no account, no loading spinner beyond 2 seconds. This means maintaining a browser-first architecture through Phase 2 and keeping file sizes optimised with WebP images and efficient code.
Pillar 04
🌍 Designed for Emerging Markets
India, Southeast Asia, and Africa are our growth markets. This means designing for lower-end devices, slower network connections, Hindi and regional language support, and pricing in local currencies. Premium pricing must feel accessible, not aspirational.
Pillar 05
💚 Sustainable Freemium Economics
The free tier must be genuinely useful and joyful — not crippled. The premium tier must be meaningfully better — not just unlocking more of the same. Revenue pays for better content, which serves the mission. We will never sell user data or introduce advertising as a revenue stream.
Pillar 06
♿ Accessibility as a Default
Keyboard navigation, screen reader support, high contrast mode, and reduced motion options are not afterthoughts — they are shipped features. A child with dyslexia or low vision deserves the same experience as any other learner. WCAG 2.1 AA is our minimum standard.
6 · Guiding Principles (Decision Rules)
🧭

What to build, what to ignore

Concrete, opinionated rules that help the team make consistent decisions — especially when stakeholders disagree. These are not aspirations; they are commitments.

01

Build for the child's delight, not the adult's reassurance.

If a feature only makes the dashboard prettier for a parent but adds friction for the child's game flow, we deprioritise it. The product's primary user is always the learner, not the payer.

02

Ship a smaller thing that works perfectly over a larger thing that works partially.

Phase 1 shipped 68 features — every single one is polished, tested, and intentional. We will never launch a half-finished feature just to hit a deadline. Quality is the brand.

03

Never use dark patterns or manufactured urgency.

No countdown timers. No "9 other parents are viewing this plan" messages. No forced interruptions mid-game to upsell. If our product is good enough, it sells itself. Trust is our longest moat.

04

If a feature needs a tutorial, redesign it.

A six-year-old and a sixty-year-old grandparent should both be able to pick up a quiz pack and play without instructions. Simplicity is not a Phase 3 polish job — it is an upfront requirement.

05

Every data decision asks: "What would happen if this appeared on the front page of a newspaper?"

Because we serve children, our data practices must be beyond reproach. Default to collecting nothing. When collection is necessary (Phase 3 accounts), make it explicit, minimal, and deletable.

06

Say no to features that serve one stakeholder at the cost of another.

Example: a mandatory daily notification might boost re-engagement numbers but would annoy parents and damage trust. Metrics that require damaging the user relationship are not valid success metrics.

Go-To-Market Strategy

GTM (Go-To-Market) is the plan for how we get Curionis in front of the right people, at the right time, with the right message. This document covers launch approach, acquisition channels, pricing, and growth milestones across all three phases.

GTM Overview · 2026–2027

From zero to 10,000 families — one pack at a time.

Curionis launches with a free, zero-friction product to build trust and a user base. Premium features are introduced in Phase 3 only after we have proof of engagement and organic demand. Our GTM is not a single campaign — it is a compounding flywheel: great content → word-of-mouth sharing → larger audience → more content investment → even better product.

GTM Philosophy & Approach
🎯

Product-Led Growth (PLG) — the core strategy

We let the product do the selling. No cold calls, no sales team, no expensive ad campaigns at launch.

What is Product-Led Growth?
PLG means the product itself is the primary driver of acquisition and retention. Users discover the product, try it for free, get genuine value, and then share it — or upgrade. This is how Duolingo, Notion, and Canva grew without traditional advertising. For Curionis, it means: a child plays the quiz → gets excited → tells a friend → friend's parent opens the link → they play → they share again. Each happy user creates the next user.
Why PLG fits Curionis
• Zero signup barrier — anyone can try in seconds
• Shareable moments — score cards, star ratings, "Did You Know" facts are all designed to be shared on WhatsApp
• Natural virality — children tell other children; parents tell other parents
• Low CAC (Cost to Acquire a Customer) — organic sharing is free
• High trust — a recommendation from a trusted friend converts better than any ad
The Flywheel
🎮 Great free product → 📤 Users share it → 👥 Larger audience → 💰 Revenue funds better content → 🔄 Even better product

Phase 2 GTM — Growth & Retention (Jun–Oct 2026)
📈

Turning one-time visitors into regular users

Phase 2 adds the tools that make Curionis something users come back to — not just try once.

PWA Install Campaign
When Phase 2 ships the Progressive Web App (PWA), users can install Curionis on their phone's home screen — like a native app, but without going to the App Store. We will prompt users to install after their second session with a friendly, skippable prompt. Target: 15% of returning users install. This turns a website visit into a persistent icon on a child's tablet — dramatically improving retention.
Score Card Sharing (Viral Loop)
The Phase 2 Score Share Card generates a beautiful image (like a sports results card) that users can share on WhatsApp, Instagram, or Twitter. Each share shows the Curionis brand, the pack name, and the score. This is an active viral loop: every share is an ad we don't pay for. Target: 2,000 shares/month by Month 3 of Phase 2.
Hindi Launch — India Expansion
Launching the full Hindi locale opens Curionis to Tier-2 and Tier-3 cities where English-first apps are inaccessible. Strategy: Announce the Hindi launch on WhatsApp education groups, reach out to Hindi-medium school principals, and create SEO content in Hindi. Expected outcome: 30–40% of new sessions from non-English-first users.
Parent Dashboard as Retention Tool
The Phase 2 Parent Dashboard lets parents see how many questions their child got right, which packs they completed, and how long they played. This gives parents a reason to return to the app themselves — and to keep their child engaged. A weekly summary email (optional opt-in) keeps Curionis top-of-mind for parents who are busy.

Key Performance Indicators (KPIs) — What We Track Weekly
📊

The numbers that tell us if GTM is working

A KPI (Key Performance Indicator) is a specific, measurable number we check regularly to see if the strategy is working. These are the eight most important ones for Curionis.

DAU

Daily Active Users

How many unique users play at least one quiz round in a day. Rising DAU = product is becoming a habit. Target: 500 DAU by end of Phase 1.

D30

Day-30 Retention Rate

The % of new users who are still active 30 days after first use. Industry average for EdTech: 20–25%. Our target: ≥25% by Phase 2.

CAC

Customer Acquisition Cost

How much we spend on average to get one new user. For organic channels this is ₹0. If we introduce paid ads in Phase 3, CAC must stay below ₹500 (our LTV is ₹3,588).

LTV

Lifetime Value per Subscriber

The total revenue one subscriber generates before they cancel. At ₹299/month with an average 12-month tenure: LTV = ₹3,588. Growing LTV requires reducing churn and increasing average tenure.

MRR

Monthly Recurring Revenue

The total predictable revenue coming in every month from active subscribers. This is the single most important financial health metric for a subscription business. Target: ₹3L MRR by end of Phase 3.

NPS

Net Promoter Score

Asks users: "How likely are you to recommend Curionis to a friend?" on a 0–10 scale. Score above 50 is excellent. This is the best leading indicator of organic word-of-mouth growth. Target: ≥60 NPS from teachers.

CVR

Free-to-Paid Conversion Rate

What % of free users upgrade to a paid plan within 30 days of premium launch. Industry benchmark for EdTech: 3–7%. We target 5% given our warm intent registration list.

K

Viral Coefficient (K-factor)

How many new users each existing user brings in through sharing. K > 1 means the product grows itself without any spend. WhatsApp sharing and score cards are our primary mechanisms. Target: K ≥ 0.5 by Phase 2.

🧭 Plain-English Guide: What does each layer actually do?

🎨 Presentation Layer

Everything the user sees — buttons, images, text, animations. Built with HTML (structure), CSS (styling), and a tiny amount of JavaScript (interactivity). Think of it as the interior design of the building.

⚙️ Logic Layer

The rules engine — "if the answer is correct, add 10 points", "if the user is a parent, show the dashboard". Written in JavaScript. This is the building's electrical wiring and plumbing — invisible but essential.

🗃️ Data Layer

Where facts, questions, images, and text live. In Phase 1 this is JavaScript files. In Phase 3 it becomes a proper database on a cloud server. Think of it as the building's library and filing rooms.

🖥️ Backend Layer (Phase 3)

A computer programme running on a cloud server that processes requests — login, payment, data sync. The child never sees it, but everything persistent (accounts, subscriptions, class data) relies on it.

🔌 External Services

Third-party companies whose specialised tools we plug into rather than build ourselves — Stripe for payments, Google Analytics for data, SendGrid for email. Like outsourcing the building's CCTV and electricity supply to specialists.

📱 Device Layer

The physical device the user holds (phone, tablet, laptop) and the software on it (browser or app). Phase 1 supports any browser. Phase 2 adds a "home screen install". Phase 3 adds native Android and iOS apps.

Before you read — key concepts explained simply

Plain-English explanations of technical terms you'll encounter throughout this document.

What is REST?
REST (Representational State Transfer) is a style of building APIs where resources (users, packs, payments) each have a URL address, and you perform operations on them using standard HTTP methods: GET (read), POST (create), PUT/PATCH (update), DELETE (remove).
What is JSON?
JSON (JavaScript Object Notation) is the data format all requests and responses use. It looks like structured text: {"name":"Curionis","version":1}. It's human-readable and universally supported.
What is JWT Authentication?
JWT (JSON Web Token) is a secure way to prove identity. After logging in, the server issues a token (a long string of characters). The app includes this token in every subsequent request — like showing a wristband at a concert to prove you have a ticket. Tokens expire after a set time for security.
Base URL & Versioning
All API endpoints are prefixed with https://api.curionis.app/v1. The /v1 version prefix means we can introduce a new API version (v2) in the future without breaking existing apps that use v1.
Rate Limiting
To prevent abuse, the API limits requests to 100 requests per minute per IP address for unauthenticated calls and 300/minute for authenticated users. Exceeding this returns a 429 Too Many Requests error. This protects server stability and prevents malicious use.
Error Format
All errors return a consistent JSON structure: {"error":"NOT_FOUND","message":"Pack not found","code":404}. Consistent error formats make debugging fast and enable the frontend to show the right user-facing message.

What this block does

Line 1 — DOCTYPE: Tells the browser "this is a modern HTML5 document". Without this, browsers use quirks mode — an old compatibility mode that renders CSS differently and inconsistently.

Line 2 — lang="en": Declares the page language as English. Screen readers use this to select the right text-to-speech voice. It also affects spell-check and translation suggestions in browsers.

Line 4 — charset UTF-8: Sets the character encoding to UTF-8, which supports every emoji and international character used in the app — essential for Hindi localisation in Phase 2.

Line 5 — viewport meta: Critical for mobile. Without this, mobile browsers render the page at ~980px wide then shrink it — making everything tiny. This tag tells them to match the device's actual screen width.

Line 6 — data-i18n-title: A custom attribute that tells the applyStrings() function to update the browser tab title from the STRINGS database. Means the tab title can be translated without touching the HTML.

Lines 7–9 — Google Fonts: Two preconnect tags tell the browser to open a connection to Google's font servers early, before it even starts downloading the fonts. This is a performance optimisation — it saves ~100ms. Line 9 loads Cormorant Garamond (the elegant serif display font) and Jost (the clean sans-serif body font). The display=swap parameter means text shows in a fallback system font immediately, then swaps to Cormorant/Jost once loaded — preventing invisible text during load.

What this file does

Single source of truth for all visible text: Every word, label, button text, error message, tooltip, placeholder, and heading that the user sees in the app is defined in this file. Changing any string here changes it everywhere in the app simultaneously — no hunting through HTML files.

var (not const or let): var STRINGS is used instead of const or let. This is deliberate — var declarations are globally scoped even when outside functions, making STRINGS available as a true global variable that app.js can access. const and let would also work here but var makes the global intent explicit.

Key naming convention: Keys follow a dot-separated hierarchy: component.section.element. Examples: "nav.home" (navbar, home link), "game.progress_label" (game page, progress text), "gate.question" (parent gate, question text). This hierarchy makes the file self-documenting and easy to scan.

Template tokens {placeholder}: Some strings contain curly-brace placeholders like {n}, {total}, {a}, {b}. These are substituted at runtime by the S(key, vars) function in app.js. The template approach means the string structure (including word order) can be adjusted per language — important for Hindi where subject-verb-object order differs from English.

Generated from Excel: The header comment says "auto-generated from curionis_translation_db_v2.xlsx". This means a Python script reads the Excel file and writes this JS object. Translators edit the Excel, not this file. This is a professional workflow — keeping source of truth in a spreadsheet that non-developers can edit.

What this file does

FACTS_DB is the knowledge layer: Where PACKS contains game mechanics (what image to show, what answers to offer), FACTS_DB contains educational content — the facts and "Did You Know" statements that appear in flashcards after each answer. Currently only the Aircrafts pack has full fact data; other packs show "Under development" when the flashcard is opened.

Three-part structure per pack: Each pack in FACTS_DB has three sub-objects: facts (a key-value record of factual fields like AKA, Maximum Speed, Manufacturer), dyk (an array of 1–5 "Did You Know" statements), and references (an array of source URLs). This structure maps directly to the two flashcard modes in the game — Facts card and Did You Know card.

Key matching by image filename: The species key (e.g., "Su-35") must match the image filename without extension (e.g., assets/images/aircrafts/Su-35.webp → key "Su-35"). The matching logic in app.js's openFlashcard() does: animalObj.img.split('/').pop().replace(/\.[^.]+$/, '') — splits the path by /, takes the last segment (filename), then strips the extension. This is the bridge between the PACKS content layer and the FACTS_DB knowledge layer.

Generated from Excel — the workflow: Curators add facts in facts_db.xlsx with separate sheets for facts, DYK statements, and references. A Python script convert_db.py reads the Excel and writes this JavaScript file. This means non-developers can add new species facts by editing the spreadsheet, then running one command. The JS file is never edited by hand.

var FACTS_DB (global): Like STRINGS, declared with var for global scope. App.js checks typeof FACTS_DB !== 'undefined' before use — graceful degradation if the file fails to load.

What this directory structure means

WebP format — why not JPG or PNG? WebP is Google's modern image format, offering 25–35% smaller file sizes than JPEG at equivalent visual quality. Smaller files mean faster loading — especially important on mobile connections in India. The cover images use WebP; Wild Animals uses PNG (possibly an older cover that predates the WebP conversion). Some wild animal filenames end in _converted.webp, indicating they were originally in another format and batch-converted.

File naming drives everything: Each image filename is the key that connects the PACKS data, the FACTS_DB, and the file system. If an image is named B-2_Spirit.webp, the PACKS array must have img: 'assets/images/aircrafts/B-2_Spirit.webp' and FACTS_DB must have the key "B-2_Spirit". Renaming an image without updating both data files breaks the facts lookup. This is why the ReadMe emphasises keeping the folder structure intact.

Folder = pack: Each subfolder under images/ corresponds directly to a pack ID in the PACKS array (e.g., folder wild-animals/ = pack id 'wild-animals'). This is enforced by convention, not code — but it makes the relationship immediately clear.

Cover images are separate: Pack cover images live in assets/covers/, not in the species image folders. They're used in the Cover Flow carousel and the home page pack preview row. Cover images are always the pack's best or most iconic image, chosen for visual appeal at the carousel card size (200×200px).

facts_db.json — the tooling mirror: facts_db.json is a JSON copy of facts_db.js. JavaScript files can't be imported by Python scripts or data tools directly, but JSON can. The Python conversion script (convert_db.py) likely reads this JSON as its template or writes both files simultaneously. Having both formats means data tools work with JSON while the browser loads JS.
